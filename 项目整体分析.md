# 通用项目代码分析工作流（中文版）

## 1. 结论（先看这个）

基于 `workflow_templates/codebase_analysis.yaml`、`workflow_templates/tools.py`、`workflow` 的解析执行代码，我给出一套可复用于任意仓库的分析流程：

1. **先做结构快照，再做证据抽样，再做并行分析，最后统一汇总与落盘。**
2. **每个分析结论都要求有文件证据（路径 + 关键片段），避免“只凭模型印象”。**
3. **通过重试、上下文压缩、并行限流与文件输出保障，让大仓库也能稳定产出。**

这套流程可以直接套在任意项目，只需要替换输入变量（`repo_path`、输出文件名、分析范围）即可。

## 3. 术语说明（首次出现即解释）

| 术语 | 含义（通俗解释） |
|---|---|
| Workflow（工作流） | 把一次分析拆成多个步骤，按顺序或并行自动执行。 |
| Agent（代理） | 执行某一类分析职责的角色（如“架构分析员”）。 |
| Parallel（并行） | 多个独立分析同时跑，缩短总时间。 |
| Output Variable（输出变量） | 把某一步结果存成变量，供后续步骤复用。 |
| Output File（输出文件） | 把某一步结果直接写入文档文件。 |
| Context Compaction（上下文压缩） | 当文本太长时自动截断/压缩，避免模型超长输入失效。 |

## 4. 可泛化到任意项目的标准流程（核心）

> 适用边界：单仓、多模块单仓（Monorepo）、多语言仓库。  
> 不适用边界：超大二进制仓库且缺乏文本源码（这类需先做文件白名单策略）。

### 阶段 A：范围定义（输入契约）

1. 定义分析目标：架构审查、模块梳理、依赖风险、质量风险。
2. 定义扫描范围：`repo_path`，必要时增加包含/排除目录（如 `src/`、`apps/`、排除 `node_modules/`）。
3. 定义输出产物：至少包含 `project_analysis.md` 与 `repository_map.md`。

### 阶段 B：仓库快照（结构证据层）

1. 用 `list_repo_dirs` 得到目录骨架。
2. 用 `list_repo_files` 得到文件清单（受 `max_items` 约束）。
3. 先形成“结构事实清单”（入口文件、配置文件、测试目录、部署目录）。

### 阶段 C：证据抽样（代码证据层）

1. 读取关键文件（`README`、主配置、入口代码、核心模块）。
2. 用 `grep_in_file` 对关键关键词定点抽样（如 `main`、`router`、`service`、`dependency`、`TODO`、`FIXME`）。
3. 所有结论必须绑定“文件路径 + 片段事实”，不允许无证据判断。

### 阶段 D：并行深度分析（职责分治）

并行四路：

1. 架构路：入口、执行链路、运行边界。
2. 模块路：职责边界、耦合关系、复用与重复。
3. 依赖路：运行依赖、版本兼容、部署约束。
4. 风险路：可维护性、可靠性、测试覆盖、可观测性缺口。

关键点：每一路都只做自己职责，减少交叉污染；然后把并行结果统一交给综合代理。

### 阶段 E：综合与文档产出（可交付层）

1. 汇总步骤把并行结果合并为 `analysis_context`。
2. 生成主报告（`project_analysis.md`）：架构、模块、依赖、风险、建议。
3. 生成结构图谱（`repository_map.md`）：目录地图、入口点、模块关系。
4. 文档必须是中文，并附“结论边界 + 未确认项 + 后续验证建议”。

### 阶段 F：质量闸门（稳定性层）

1. 文档非空检查（长度 > 0）。
2. 章节完整性检查（至少含：架构、模块、依赖、风险、建议）。
3. 失败自动重试（`max_retries` + `retry_delay`）。
4. 遇到空响应时利用执行器兜底能力继续落盘，保证流程不断。

## 5. 与 `codebase_analysis.yaml` 的映射关系（如何直接套用）

| 通用阶段 | 模板位置 | 已具备能力 | 建议增强 |
|---|---|---|---|
| B + C（快照/抽样） | `parallel_analysis` 前置上下文 | 当前由各代理自行调用工具 | 增加一个显式“预扫描步骤”，先产出统一事实基线 |

## 6. 推荐的“任意项目”使用方式（SOP）

### 6.1 最小执行命令（Windows PowerShell）

```powershell
D:\conda\envs\praisonai\python.exe -m praisonai workflow run .\workflow_templates\codebase_analysis.yaml `
  --var repo_path="E:/your-project" `
  --var output_doc="project_analysis.md" `
  --var output_map="repository_map.md" `
  --max-tokens 16000 -v
```

### 6.2 结果验收命令

```powershell
Get-Item .\project_analysis.md, .\repository_map.md | Select-Object Name,Length,LastWriteTime
Get-Content .\project_analysis.md -Head 30
Get-Content .\repository_map.md -Head 30
```

## 8. 风险与边界（深度思考后的约束）

1. **工具层是“文本证据优先”**：二进制、生成产物、超大日志不适合直接纳入主上下文。
2. **并行不是越多越好**：第三方网关不稳定时，需限制并行 worker（已有 `PRAISONAI_WORKFLOW_PARALLEL_MAX_WORKERS` 支持）。
3. **“能跑完”不等于“分析可靠”**：必须把“结论-证据”对齐作为验收标准。
4. **模型空响应是现实问题**：当前执行器有空响应重试与兜底机制，但仍建议保留人工 spot check（抽样复核）。

---

## 9. 最终建议

如果你要把这套流程长期用于“任意项目代码分析”，建议固化三件事：

1. 保留当前并行四路结构（架构/模块/依赖/风险），这是性价比最高的职责拆分。
2. 在模板里增加“预扫描步骤 + 最终自检步骤”，让证据链更稳定。
3. 把中文输出规范、术语解释规范、结论边界规范写进 `synth_agent.instructions`，减少报告风格漂移。

这样，你的工作流就不是一次性脚本，而是可持续复用的“代码库分析标准作业流”。
