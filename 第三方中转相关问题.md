# 第三方中转站相关问题 FAQ

> 本文档整理使用第三方 OpenAI 兼容中转站时的常见疑问，基于本项目实际排障经验总结。

---

## Q1: 为什么 workflow 路径要显式传 `max_tokens`？

因为 workflow 路径和 direct 路径的参数组装是**两条独立链路**：

- **direct 路径**（`praisonai "任务"`）：CLI 会自动设置默认 `max_tokens=16000`
- **workflow 路径**（`praisonai workflow run`）：以前没有透传这个参数，发给网关的是 `null`

`max_tokens=null` 的含义对不同端点不一样：

- 官方 API 收到 `null`：有合理的内部默认值，正常返回
- 部分第三方网关收到 `null`：可能理解为"不生成任何 token"，直接返回 `finish_reason=stop` + `content=None`

这就是"同一个模型同一个网关，direct 能出结果，workflow 空响应"的根因之一——不是模型不行，而是 workflow 少传了一个参数。

修复本质是**参数对齐**，让两条路径发出去的请求一致。对官方 API 来说加不加都没影响，但对第三方网关是必要的。

## Q2: 很多中转站不就是直接中转官方 API 吗？为什么会有这么多兼容问题？

理想中的"中转站"应该是透明代理，但实际上大多数第三方网关**不是纯中转**，而是"协议翻译层"。原因有三：

### 2.1 很多网关背后不是官方 OpenAI

虽然对外暴露的是 OpenAI 兼容接口（`/v1/chat/completions`），但背后可能是：

- 多个 provider 的负载均衡（OpenAI + Claude + 开源模型混用）
- 自建推理集群（vLLM、TGI 等）
- 二次封装的中间层（做计费、限流、审核、缓存）

这些中间层各自重新实现了 OpenAI 的 API 协议，但**只实现了子集**。`max_tokens=null` 在官方 SDK 里有明确语义（"用模型默认值"），但一个自己写的网关可能就是把 `null` 原样传给后端推理引擎，而后端引擎的行为未必一样。

### 2.2 即使是纯中转，也会做请求改写

典型操作：

- 把 `gpt-5.3-codex` 映射到实际可用的模型（可能是另一个版本）
- 剥离或改写不认识的参数（如 `tool_choice`）
- 把 Responses API 请求降级为 Chat Completions
- 对请求/响应做 JSON schema 校验，格式不符直接拒绝或返回空

### 2.3 协议版本差异

OpenAI 的 API 一直在演进（`function_call` -> `tool_calls` -> `responses`）。网关实现时锚定的是某个时间点的协议版本，之后新加的字段或语义变化不一定跟进。`max_tokens=null` 在旧版协议里可能就不是合法值。

### 结论

**"OpenAI 兼容"不等于"OpenAI 透传"。** 大部分网关是用自己的代码重新实现了这套协议，实现深度参差不齐，所以同一个请求参数在不同端点的行为会不一样。

## Q3: 同一个模型，不同中转站问同一个问题，结果可能不一样吗？

对，而且差异可能比预期大。不只是"回答措辞不同"（那是模型本身的随机性），而是**结构性差异**：

| 层面 | 具体表现 |
|---|---|
| 模型版本 | 你以为调的是 `gpt-4o`，但中转站可能映射到了旧版 snapshot，甚至是另一个模型 |
| 参数改写 | 中转站可能静默修改了 `temperature`、`max_tokens`、`top_p` 等参数 |
| 系统提示注入 | 有些中转站会偷偷加 system prompt（做审核、品牌露出等），改变模型行为 |
| 响应截断 | 中转站自己有 token 限制，可能比官方更短，导致输出被截断 |
| 工具调用 | 有的支持、有的不支持、有的只支持旧版 `function_call` 格式 |
| 缓存命中 | 有些中转站做了语义缓存，相同问题直接返回之前的结果，不再调模型 |
| 内容过滤 | 各家审核策略不同，同一个问题在 A 正常回答，在 B 被拒绝或删减 |

本项目文档 16.x 节反复排查的本质就是在应对这个现实：**你无法假设中转站的行为和官方一致**。每换一个中转站，上面这些差异的组合就可能不同，表现出来就是"同样的代码，换个 API 地址就不工作了"。

## Q4: 是不是不是所有中转站都能在本项目正常调用？

对。能不能正常工作取决于中转站实现了多少协议特性。

### 本项目的最低要求

| 能力 | 原因 |
|---|---|
| `POST /v1/chat/completions` | 基础对话接口 |
| `tool_calls` 字段（非旧版 `function_call`） | workflow 里的工具调用全靠这个 |
| `max_tokens` 参数正常处理 | 不能收到就返回空 |
| 流式响应（`stream=true`） | verbose 模式和部分交互路径依赖 |

### 如果中转站缺了某项

| 缺失能力 | 后果 | 本项目的兜底 |
|---|---|---|
| 不支持 `tool_calls` | 工具完全不可用，agent 只能"说"不能"做" | 文本 JSON fallback 解析，但不保证稳定 |
| `max_tokens=null` 返回空 | workflow 空响应 | 已自动透传默认值 |
| 流式里不带 `delta.tool_calls` | 工具调用丢失 | 已放开门控，但中转站完全不返回就没办法 |
| 静默改写模型/参数 | 行为不可预测 | 无法兜底，只能靠用户自己验证 |

### 换中转站时的冒烟测试（推荐）

```powershell
# 1. 基础对话（无工具）
praisonai "只回复OK" --llm custom_openai/你的模型 --no-tools

# 2. 工具调用
praisonai "列出当前目录文件" --llm custom_openai/你的模型

# 3. workflow
praisonai workflow run .\workflow_templates\codex_agent_orchestration\codex_two_step_smoke.yaml -m custom_openai/你的模型
```

三步都通过才说明这个中转站和本项目兼容。任何一步失败，就需要看日志定位是哪个能力缺失。

## Q5: 这些第三方兼容修复会影响官方 API 的正常调用吗？

不影响。所有修复都是**条件性的**，只在检测到第三方端点时才生效。判断逻辑是通过 `base_url` 是否包含官方域名（`api.openai.com`、`openai.azure.com` 等）来区分。

| 修复项 | 官方 API 下的行为 |
|---|---|
| 模型名加 `custom_openai/` 前缀 | 不加，保持原名 |
| `_supports_responses_api()` 返回 `False` | 保持返回 `True` |
| 移除显式 `tool_choice='auto'` | 保持注入 |
| 内部参数过滤（`force_tool_usage` 等） | 同样过滤，官方 SDK 也不认这些参数，过滤掉更安全 |
| 文本提取扩展候选字段 | 优先读 `message.content`，找到就返回，不会误读非标字段 |
| 空响应重试 | 官方 API 正常返回时不触发重试条件 |
| `max_tokens` / 模型名透传 | 对官方 API 同样有效，属于 bug 修复而非兼容 hack |

官方 API 路径上这些代码要么不被执行，要么行为与修改前完全一致。
